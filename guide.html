Companion Guide

How to understand and extend the website

The homepage is now the main product: it already includes a built-in BI dashboard, a project summary, the schema section, and links to the SQL deliverables. This guide explains how each part is meant to be used.

Site Structure

What each page is for

Home page

The main website presents the project story and contains the built-in interactive dashboard.

Guide page

This page explains the structure, the intention behind the sections, and the optional future upgrades.

SQL files

The SQL scripts remain the formal technical deliverables for schema creation and analysis logic.

Dashboard

What the built-in BI section includes

KPI cards

Show repositories, developers, stars, and commits for the current filtered view.

Charts

Show top repositories, developer rankings, language mix, impact distribution, star-count buckets, lifecycle views, and quality-related summaries.

Repository spotlight

Highlights the top filtered repositories instead of dumping the entire dataset into one large table.

Power BI Version

How it fits beside the built-in dashboard

Built-in website dashboard

Best for GitHub Pages, quick browsing, and keeping the portfolio fully self-contained.

Formal Power BI report

Best for a standard BI deliverable, classroom presentation, and interviews where you want to show Power BI specifically.

Recommended setup

Keep both. Use the website as the main portfolio entry point and add a published Power BI link as an alternate version.

Optional Next Step

What you can add later

Power BI version

You can still build a Power BI report later and either link to it or embed it as an additional page. The fastest starting point is docs/powerbi-quickstart.md.

Full dataset already prepared

The repository-wide BI file is already available in data/repository_dashboard_full.csv.

GitHub Pages deployment

Once GitHub Pages is enabled on the `main` branch, this site becomes your live portfolio URL.
