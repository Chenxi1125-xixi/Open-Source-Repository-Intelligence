Project Positioning

What makes this stronger than a class project

Relational modeling

The project is not just query practice. It includes a normalized 5-table schema with PK/FK constraints, indexing, lookup tables, and a reusable analysis view.

Data quality decisions

It addresses realistic issues including sparse owner matching across datasets, text-to-numeric cleaning for commit fields, and language representation bias.

Executive-friendly outputs

The SQL layer is packaged into a Power BI narrative that helps hiring managers quickly understand the analytical question, the data model, and the business takeaway.

Architecture

From denormalized exports to reusable analytics model

  1. Collect public GitHub users and repository metadata from Kaggle.
  2. Resolve developer-owner relationships and clean free-text metrics.
  3. Load data into normalized MySQL tables for developers, repositories, metrics, companies, and languages.
  4. Create reusable SQL views and feed the final analysis table into Power BI.

Insights

Core findings worth highlighting in interviews

Activity is concentrated

A small number of organization-backed repositories account for most of the star and fork volume. In this sample, IBM Cloud and AWS dominate visibility metrics.

Most repos are still early stage

44 of 55 repositories fall into the early-stage bucket, which makes the high-impact repositories easier to spot and compare.

Long-lived projects stand out

Repository lifespan analysis reveals a few projects with exceptionally long active windows, useful for studying sustained maintenance.

Schema design enables reuse

The final analysis view reduces repeated joins and makes the BI layer simpler to maintain, which is a strong engineering talking point.
